# 運用・保守手順書 (Operations)

本システムの日常のデータ更新、システムメンテナンス、定期点検、およびバックアップに関する運用手順を定めます。

---

## 1. データの追加・修正・削除フロー

掲載事業所データの更新は、原則として [facilities.csv](file:///c:/Projects/care-portal_v2/data/facilities.csv) を直接編集することで行います。

### 更新手順
1. **CSVの編集**:  
   マニュアルに従い、Excelやテキストエディタ等で [facilities.csv](file:///c:/Projects/care-portal_v2/data/facilities.csv) にデータを追加・編集します。
   * ※行追加時のID体系や丁目の正規化ルールについては [ADD_CITY.md](file:///c:/Projects/care-portal_v2/ADD_CITY.md) を参照してください。
2. **ローカル検証**:  
   ```bash
   npm run dev
   ```
   ローカルでWebサーバーを起動し、追加したデータが正しく絞り込めるか、および詳細ページ（`/facilities/facility-XXXX`）が表示されるか確認します。
3. **ビルドテスト**:  
   ```bash
   npm run build
   ```
   エラーなく静的生成が完了することを確認します。これにより、パースエラーや静的パス生成（slugの重複等）のバグを事前に防ぎます。
4. **反映・デプロイ**:  
   変更をGitにコミットし、GitHubへプッシュします。GitHubにプッシュされると、Vercelなどの接続プラットフォームで自動的にビルド＆デプロイが行われ、本番環境に反映されます。

---

## 2. 定期点検 (Periodic Inspection)

本ポータルを健全に保つため、半年に1回を目安に以下の定期点検を実施します。

1. **データ不整合チェック**:
   * 重複する `slug` や `officeNumber` が登録されていないか。
   * 公開フラグ（`isPublished`）が `true` にもかかわらず必須項目（`name`, `type`, `area`, `address`, `tel`）が欠落していないか。
2. **外部リンクの生存チェック**:
   * 各事業所詳細ページに掲載されている公式サイト（`website`）のリンク切れがないか。
3. **ビルド性能の測定**:
   * 静的生成ページ数（現在は約1,500ページ）の増加に伴い、ビルド時間やTurbopackの実行時間が許容範囲内であるか。

---

## 3. バックアップ方針 (Backup Policy)

* **Gitでの履歴管理**:  
  マスタデータである [facilities.csv](file:///c:/Projects/care-portal_v2/data/facilities.csv) はリポジトリ内で管理されているため、過去のすべての状態はGitコミット履歴から復元可能です。これが一次バックアップとなります。
* **手動スナップショット**:  
  一括データ流し込みや大幅なデータ更新を行う前には、必ず手動でCSVファイルのバックアップを作成してください。
  * バックアップファイルは `data/facilities_backupXXX.csv` という名称で同一フォルダに保存されています。
  * データの更新が正常に完了し、動作に問題がないことを確認したのち、不要になった過去数世代前より古いバックアップファイルは整理（削除）してください。

---

## 4. 運用ルール (Rules)

* **カンマ `,` の使用禁止**:  
  CSVファイルであるため、説明文（`desc`）や住所（`address`）の中に半角カンマ `,` を使用しないでください。パース崩れの原因となります。
* **文字コードの厳守**:  
  CSVファイルは必ず **UTF-8 (BOMなし)** 形式で保存してください。Excelで上書き保存する際は特に文字コードの変更（Shift-JISへの勝手な変換等）に注意してください。
