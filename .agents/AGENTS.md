# H3 Incover Development Standard v2.0

本プロジェクトにおける開発・運用・保守の方針です。技術負債、理解負債、認知負債を防ぎ、誰でも継続して開発・改善できる仕組みを維持するためのルールです。

---

## 基本方針

1. **設計を先に考える**
   * 実装よりも設計を優先します。目的・役割・データの流れ・影響範囲をドキュメント等で整理したうえで実装を開始してください。
2. **コードだけを成果物にしない**
   * コード変更に加え、その設計判断の理由（なぜそうしたか）、運用方法、保守方法を成果物（`docs/`）としてドキュメントに残してください。
3. **運用まで設計する**
   * 単発の機能追加ではなく、継続してアップデート・保守できる仕組み・プロセスを意識して設計してください。
4. **理解目標**
   * コードの細部を暗記することではなく、以下の項目を他の開発者やAIに対して「説明できる状態」にすることを目指します。
     * この機能の目的
     * データの流れ
     * 関連ファイル
     * 変更した理由
     * 将来変更するときの注意点

## 必須ドキュメント構成

仕様変更・機能追加時は、必要に応じて `docs/` 配下の以下のファイルを更新してください。

* **[docs/vision.md](file:///c:/Projects/care-portal_v2/docs/vision.md)**: プロジェクトの目的・理念・提供する価値・AI活用方針。
* **[docs/architecture.md](file:///c:/Projects/care-portal_v2/docs/architecture.md)**: システムの全体構成、データの流れ、主要ファイルの役割。
* **[docs/decisions.md](file:///c:/Projects/care-portal_v2/docs/decisions.md)**: 設計における選択肢の比較、決定理由、将来変更を検討する基準（トリガー）。
* **[docs/changelog.md](file:///c:/Projects/care-portal_v2/docs/changelog.md)**: 機能・データ追加や不具合修正、およびドキュメント更新の履歴。
* **[docs/operations.md](file:///c:/Projects/care-portal_v2/docs/operations.md)**: マスタデータの更新手順、バックアップ方針、定期保守・点検ルール。
* **[docs/data-update-policy.md](file:///c:/Projects/care-portal_v2/docs/data-update-policy.md)**: 介護事業所データの追加・修正・削除プロセスおよび段階的な非公開化ルール。
* **[docs/features/](file:///c:/Projects/care-portal_v2/docs/features/)**: 各機能（検索、ニュースなど）の目的・仕様・データフロー解説。

## 完成の定義（Definition of Done）

以下の項目がすべて揃った時点で、タスクの「完了」と定義します。

1. **プログラムコードが意図通りに動作する。**
2. **関連ドキュメント（上記必須ファイル群）が最新の状態に更新されている。**
3. **運用方法や保守フローがドキュメントで明確に定義されている。**
4. **将来の変更契機や変更方針が `docs/decisions.md` 等に記載されている。**
5. **後続のエンジニアや他のAIが引き継いで理解できる状態になっている。**
