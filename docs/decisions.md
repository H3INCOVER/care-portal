# 重要な設計判断と合意事項 (Decisions)

本プロジェクトにおける重要な技術選定・設計判断、その理由、他案との比較、および「将来どの条件に達したら再設計を検討するか（トリガー）」についてまとめます。

---

## 1. CSVファイルを主データベースとする選定

* **判断内容**:  
  データベースサーバーや外部CMSを使わず、[facilities.csv](file:///c:/Projects/care-portal_v2/data/facilities.csv) をアプリケーション内マスタデータとして採用しました。
* **他案との比較**:
  * **案A：ヘッドレスCMS（MicroCMSなど）の導入**
    * *長所*: UI上で簡単にデータ登録・編集ができる。
    * *短所*: API通信のオーバヘッドがあり、データ量が多くなった際に無料枠制限を超えてランニングコストが発生する。
  * **案B：マネージドデータベース（PostgreSQL等）の導入**
    * *長所*: SQLでの高度なリレーションやリアルタイム更新が容易。
    * *短所*: サーバーホスティング費用がかかり、管理画面を別途実装する必要がある。
  * **本案（CSVマスタ）**: 最もシンプルで、Gitにデータ変更履歴がすべて残り、運用コストが「完全無料」となるため採用。
* **将来変更を検討する条件（トリガー）**:
  * **条件1**: 事業所数が **10,000件** を超え、CSVパースに伴うビルド速度の大幅な悪化（Vercelでのビルド時間が5分超過）がみられた場合。
  * **条件2**: 非エンジニアがリアルタイムでデータを安全に頻繁更新する必要が生じ、Git経由のデプロイフローが運用に合わなくなった場合。
  * ➡ 上記のいずれかに達した場合は、SQLite（ローカルDB化）またはヘッドレスCMSへの移行を検討します。

## 2. トップページ「エリアから探す」の市区町村単位への統一

* **判断内容**:  
  当初は全詳細エリア（地区）を直接トップページに並べていましたが、掲載自治体の増加に伴いボタンが乱立するため、トップページは「市」単位（福岡市、北九州市、春日市）のボタンに統一しました。
* **他案との比較**:
  * **案A：全地区名（約50以上）をトップページにそのまま平坦に羅列する**
    * *短所*: 画面がボタンで埋め尽くされ、スクロール量が増え、ユーザーの認知負荷が著しく高まる。
  * **案B：福岡市・北九州市のみ「区」単位を残し、春日市は「市」単位にする**
    * *短所*: 階層構造がバラバラになり、プログラムの絞り込みロジックが複雑化・特殊化する。
  * **本案（市単位に統一）**: どの自治体でも表示構造が「市」で揃い、統一されたURL規則（`/facilities?city=〇〇`）で処理できるため採用。
* **将来変更を検討する条件（トリガー）**:
  * **条件1**: 掲載対象の「市区町村」の数自体が **20件** を超え、トップページに市ボタンを並べるだけでも画面が煩雑になった場合。
  * ➡ その場合は、地域ブロック（例：福岡地方、筑後地方など）でのアコーディオンUIの導入や、市区町村を選択する検索フォーム/ドロップダウンUIへの移行を検討します。

## 3. エリアデータの「丁目」除去（正規化）

* **判断内容**:  
  春日市などのデータを追加する際、元の公開データ（BODIK等）に含まれる `春日市小倉一丁目` `春日市小倉四丁目` などのエリア表記から「丁目」をプログラムおよびCSV側で除去し、`春日市小倉` のように統一しました。
* **他案との比較**:
  * **案A：丁目情報をそのまま保持して検索対象・エリア選択肢にする**
    * *短所*: 「小倉一丁目」「小倉二丁目」など非常に狭い地域ごとにフィルターボタンが生成され、フィルターの数が膨大になる。
  * **案B：表示上だけで「丁目」をトリムし、CSVの中身はそのままにする**
    * *短所*: ページ内での検索ロジックやパンくず表示で、常に表示とマスタデータの間で変換が必要になり、プログラムに複雑なロジックが残る。
  * **本案（CSVデータ自体の正規化）**: データソースの時点で正規化されているため、Next.jsの表示側（フィルター生成など）は完全にシンプルになり、認知・理解負債を抑えられるため採用。
* **将来変更を検討する条件（トリガー）**:
  * **条件1**: 特定の「丁目」を明確に区別して探したいというユーザーからの強い要望が発生した場合。
  * **条件2**: 丁目を除去して集約した結果、その地区（例：小倉）に属する事業所が **100件** を超え、さらに細かく絞り込む必要が生じた場合。
  * ➡ その場合は、エリア内の「詳細絞り込み（チェックボックスなど）」としての二次フィルターの設計を検討します。
